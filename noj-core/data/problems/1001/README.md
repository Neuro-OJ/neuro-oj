# 1001 T0-LMCC：星港舱门报码归一化

## 题目背景

星港是一座用于向太空输送物资的集散港。运行区有一整套连接主压差环、减压环廊、货运气闸和外作业桥的标准舱门网络。任何一扇舱门报码不清，都会影响人员隔离、货箱转运和局部舱段的气压切换。

一线报码通常来自巡检蜂群、拖运司机和夜班值守员，短、急、乱，但主控台需要的是稳定、可解析、可落库的标准 JSON。你的任务是通过 Prompt 约束，让模型把这些自然语言报码整理成统一结构。

## 题目描述

对于每条输入报码，模型最终必须输出一个 JSON，包含两个字段：

1. `gate_id`：标准化舱门编号
2. `status`：标准化舱门状态

`gate_id` 的格式固定为 `X-YY`：

- `X` 为区域字母，只能是 `E / W / N / S / I / O`
- `YY` 为两位数字编号

区域映射规则如下：

- `E`：东环 / 东区 / 东侧
- `W`：西环 / 西区 / 西侧
- `N`：北环 / 北区 / 北侧
- `S`：南环 / 南区 / 南侧
- `I`：主环 / 内环 / 内侧
- `O`：外环 / 外侧

编号规则如下：

1. 需要从文本中识别"几号门"。
2. 阿拉伯数字直接转为两位数字，例如 `2号门` -> `02`。
3. 中文数字也要识别，例如 `七号门` -> `07`，`十号门` -> `10`。

`status` 只能是以下三种之一：

- `open`
- `closed`
- `fault`

状态按以下优先级判定：

1. 若文本中出现故障语义，如"故障""打不开""拉不开""失灵""卡住""异常""坏了"，则判为 `fault`。
2. 否则，若出现关闭语义，如"关闭""封闭""锁住""关着""暂停通行"，则判为 `closed`。
3. 否则，若出现开启语义，如"开启""打开""恢复通行""放行""通了""重新打开"，则判为 `open`。

也就是说：`fault > closed > open`。

## 你需要实现

在 `submission.py` 中完成：

1. `build_messages(raw_text: str) -> list[dict]`
2. `build_generation_parameters() -> dict`

## 输入输出样例

### 样例输入 1

```
外环七号门完全拉不开
```

### 样例输出 1

```json
{"gate_id":"O-07","status":"fault"}
```

### 样例输入 2

```
主环8号门已经放行
```

### 样例输出 2

```json
{"gate_id":"I-08","status":"open"}
```
